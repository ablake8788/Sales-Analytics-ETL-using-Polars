WHAT THIS PROJECT IS
---------------------
Reads raw Schwab BUY/SELL trade rows from SQL Server (dbo.SchwabTransactions),
uses Polars to FIFO-match each SELL against prior BUY lots per symbol, computes
realized gain/loss + holding period + short/long-term classification, rolls the
results up by symbol x month/year, and optionally writes both levels back to
SQL Server.

Standalone sibling to schwab_market_data and schwab_market_data_pyspark. Never
modifies either. Its own output tables (SchwabSalesAnalytics_RealizedGains,
SchwabSalesAnalytics_Summary) are separate from both of those projects'
analytics tables.

STEP 3 — How FIFO matching works (etl/fifo_matching.py)
#################
For each symbol: BUY rows are pushed onto a queue in trade-date order. Each
SELL consumes from the front of the queue (oldest lots first), splitting
across multiple BUY lots if one lot isn't enough. Commission/fees on both
sides are allocated proportionally to the matched quantity. Holding period
= SellDate - BuyDate; more than 365 days => LONG, otherwise SHORT.

If a SELL's quantity exceeds everything available in the BUY queue (e.g.
missing purchase history), the shortfall is emitted as a trailing row with
IsUnmatched = true and the financial columns left null — surfaced with a
log.warning() and a console warning in main.py, rather than silently
dropped or raising.

etl/summary.py rolls matched (non-unmatched) rows up to Symbol x Period
totals: TradeCount, TotalProceeds, TotalCostBasis, TotalRealizedGainLoss,
ShortTermGainLoss, LongTermGainLoss.

IMPORTANT CAVEATS on that piece (read before relying on it):
  1. Trader API vs Market Data API — the Trader API (accounts/transactions)
     is a SEPARATE, separately-approved Schwab product from the Market Data
     API schwab_market_data already used for quotes. If the app registration
     only has Market Data approved, get_account_numbers() will 401/403.
  2. Live token check performed this session: schwab_tokens.json's refresh
     token was tested against Schwab's live token endpoint and confirmed
     EXPIRED/INVALID (invalid_grant — refresh tokens last ~7 days; the file
     was from April). A fresh interactive browser login is required before
     any live call can succeed.
  3. transaction_parser.py's field-mapping for Schwab's `transferItems`
     schema was written against Schwab's documented Trader API shape, NOT
     verified against a real live response (no valid token was available
     to fetch one). transactions_main.py prints the first raw transaction
     it fetches before parsing, specifically so this can be sanity-checked
     on the first real run.
  4. Per the assistant's own operating rules, it does not perform
     interactive OAuth logins / enter credentials on the user's behalf —
     that step is the user's to run, in their own terminal.
